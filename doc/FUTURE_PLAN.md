# Minimalist New Tab 后续开发计划

## 目标

这份文档用于指导后续版本的迭代，重点覆盖两条主线：

1. 功能继续丰富，但不破坏“轻量、零依赖、无构建”的项目特性。
2. 设计风格继续收敛，从“可用的玻璃拟态新标签页”进化到“有明确视觉系统的成品化界面”。

当前项目已经具备完整的基础能力，但后续开发需要注意一个前提：
核心实现仍然较集中，主要逻辑在 `app.js`，主要样式在 `newtab.html`。在继续加功能之前，应适度整理结构，否则每次视觉和交互调整的成本会越来越高。

## 版本路线

### Phase 1: 稳定和收口

目标：解决当前体验中的边角问题，统一视觉系统，降低后续开发成本。

建议事项：

- 清理搜索区、书签区、Dock、设置侧栏之间的视觉不一致。
- 统一表面层规则：背景、边框、阴影、圆角、焦点态、悬停态。
- 继续排查 Chromium 下的合成层、滤镜、hover 重绘问题，优先稳定而不是追求更强的特效。
- 把当前的设计变量梳理成更明确的 token，例如：
  - 表面层
  - 文字层级
  - 强调色
  - 边框强度
  - 阴影层级
  - 动效强度
- 整理 `app.js` 内 UI 管理、设置管理、搜索管理的职责边界。
- 减少“改一个 hover 影响整块界面”的情况。

建议交付结果：

- 一个稳定的默认主题
- 一套统一的按钮/卡片/面板规范
- 一版桌面端和移动端都更稳定的布局

### Phase 2: 设计系统化

目标：把当前界面从“局部好看”升级为“整体一致”。

建议事项：

- 设计 3 到 4 套明确主题，而不只是参数预设。
- 每套主题不仅修改尺寸，还应修改：
  - 表面透明度
  - 文本对比
  - 强调色
  - 阴影密度
  - 动效力度
- 建立一套完整的视觉规范：
  - Hero 区规范
  - 搜索区规范
  - 书签卡片规范
  - Dock 规范
  - 设置侧栏规范
  - 弹窗规范
- 规范 favicon 展示策略，避免不同网站图标造成整体风格破碎。
- 为弱对比背景、亮色背景、复杂背景准备不同的可读性策略。

建议主题方向：

- `Focus`
  更克制、更稳，适合办公。
- `Glass`
  保留玻璃感，但收敛高光和 glow。
- `Dense`
  适合信息密度更高的书签用户。
- `Immersive`
  背景沉浸感更强，但仍然优先保证文字可读性。

### Phase 3: 功能扩展

目标：在不破坏主页简洁感的前提下，增加高频、实用、可关闭的功能。

建议优先级如下。

高优先级：

- 书签收藏夹置顶
- 最近访问
- 常用网站自动推荐
- 多套 Dock 配置切换
- 搜索快捷命令
  - `/` 搜书签已经有基础
  - 后续可扩展 `g`、`yt`、`gh`、`ai` 等快捷前缀
- 书签卡片显示模式切换
  - 仅图标
  - 图标加标题
  - 紧凑模式

中优先级：

- 待办或轻提醒模块
  - 必须可关闭
  - 不能破坏主页主视觉
- 便签卡片
- 日历小组件
- 天气组件
- 自定义欢迎语
- 多搜索引擎一键切换栏

低优先级：

- 数据看板类模块
- 多栏小组件布局
- 复杂信息流

低优先级的原因不是没价值，而是它们非常容易破坏当前项目的简洁定位。

### Phase 4: 个性化与同步

目标：让用户的配置更有长期价值，而不是只能本地玩一次。

建议事项：

- 支持主题导出与分享
- 支持壁纸方案包
- 支持多套首页布局配置
- 支持云端同步
  - 如果保持轻量路线，可先只做本地文件同步方案
- 支持更完整的书签个性化：
  - 自定义图标
  - 自定义分组封面
  - 自定义颜色标签

## 设计优化重点

### 1. 重新定义视觉层级

当前页面的重点区域包括：

- 时间和日期
- 搜索框
- 面包屑和操作按钮
- 书签区
- Dock

后续应明确主次关系：

- 第一视觉中心：时间和搜索
- 第二视觉中心：当前书签路径和操作
- 第三视觉中心：书签内容
- 固定辅助层：Dock

不建议让 Dock、工具按钮、搜索框同时使用过强的高光和阴影，否则会互相抢注意力。

### 2. 提高背景适应能力

背景图是这个项目的氛围核心，但也是可读性风险来源。

后续应重点优化：

- 不同背景亮度下的自动对比策略
- 顶部和底部的渐变遮罩
- 背景复杂区域下的文本保护
- 壁纸类型差异化策略
  - 风景
  - 插画
  - 低对比纯色
  - 高细节人像

建议后续加入“背景分析后自动调整覆盖层”的能力，而不是完全依靠固定透明度。

### 3. 优化图标体系

当前视觉完成度很大程度取决于 favicon 质量。

后续建议：

- 为低质量 favicon 提供统一底板
- 为缺失 favicon 提供更精致的 fallback
- 为文件夹提供更统一的图标样式
- 提供站点图标裁切策略
- 控制圆角、留白和缩放规则

设计目标不是“尽量显示原 favicon”，而是“让图标系统整体协调”。

### 4. 收紧动效

当前项目适合使用轻量、短时、明确目的的动效。

建议保留：

- 页面初次进入时的淡入
- 搜索建议弹出
- Dock 拖拽反馈
- 模态框进入退出

建议避免：

- 大范围 glow 扩散
- 复杂的 hover 外溢
- 大面积滤镜联动
- 重度依赖 backdrop-filter 的交互效果

## 技术重构建议

后续如果要继续迭代，建议优先做小规模结构整理，而不是继续把所有逻辑堆在现有文件里。

建议拆分方向：

- `app.js`
  - state
  - storage
  - search
  - bookmark editor
  - dock drag
  - settings
  - ui rendering
- `newtab.html`
  - 基础 token
  - layout
  - search
  - bookmark grid
  - dock
  - modal
  - sidebar

即使不引入构建工具，也可以用更清晰的模块边界来降低维护成本。

## 风险与约束

后续开发时应始终注意以下约束：

- 保持零依赖和无构建的开发体验
- 避免引入会显著拖慢首屏的复杂组件
- 避免过度依赖滤镜和大面积实时模糊
- 不要让主页变成信息流或控制台
- 每新增一个模块，都应该允许关闭或隐藏
- 先保证默认体验好，再提供深度可定制

## 建议的近期任务清单

适合下一阶段直接落地的事项：

1. 整理搜索区的视觉实现，彻底规避 hover 和滤镜伪影。
2. 统一按钮、面板、卡片、弹窗的视觉 token。
3. 为书签卡片建立更稳的图标展示规则。
4. 完善中屏和移动端布局。
5. 增加一套真正意义上的主题系统，而不是纯参数滑块。
6. 把 `app.js` 中与 UI 绘制无关的逻辑进一步拆开。

## 设计验收标准

后续每次设计优化可以按下面的标准自检：

- 不同壁纸下，时间和搜索是否仍然清晰
- 搜索框 hover、focus、菜单展开时，是否不会污染下方内容
- 书签第一屏是否主次分明、视觉稳定
- Dock 是否存在感足够但不过度抢眼
- 设置面板是否像“产品设置”而不是“开发调参面板”
- 桌面端、窄窗口、移动端是否都成立
- 页面是否仍然保持快速、轻量、无明显卡顿

## 结论

后续开发最重要的原则不是“继续加功能”，而是：

- 先把视觉系统做稳定
- 再把结构做清楚
- 最后再扩展高频功能

如果这三个顺序反过来，后面每一次调整成本都会更高。
