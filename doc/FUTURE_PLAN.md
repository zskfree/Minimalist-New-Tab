# Minimalist New Tab 后续开发计划

## 计划定位

这份计划用于指导后续版本迭代。当前项目已经不是早期原型，而是一个可用的 Chrome MV3 新标签页扩展，已有能力包括：

- 壁纸来源切换：Bing、Anime、Picsum、自定义 URL、本地图片。
- 搜索引擎切换、搜索建议代理、搜索历史。
- Chrome 书签同步、书签编辑、新增、删除、自定义图标、HTML 导入。
- Dock 快捷入口、拖拽排序、移动端长按拖拽。
- 样式滑块、快速预设、配置导入导出、中英文文案。
- 保持零依赖、无构建、直接加载源码的开发方式。

因此后续开发的核心不是继续堆功能，而是把现有能力产品化：

1. 先稳定视觉、交互和性能。
2. 再拆清代码边界，降低维护成本。
3. 最后只扩展真正高频、可关闭、不会破坏主页简洁感的功能。

## 当前项目判断

### 合理之处

原计划的大方向是合理的：

- 先做视觉收口，再做功能扩展。
- 坚持轻量、零依赖、无构建。
- 关注背景图下的可读性、favicon 质量、动效克制。
- 意识到 `app.js` 和 `newtab.html` 已经过于集中。

### 需要调整之处

原计划的问题主要是粒度偏大，部分功能没有区分实现成本和权限成本：

- `最近访问`、`常用网站自动推荐` 需要 `history` 或 `topSites` 权限，会影响用户信任，应单独评估，不能作为普通功能项直接排进高优先级。
- `天气组件`、`日历小组件`、`数据看板` 会把主页从新标签页工具推向信息面板，和当前定位冲突，默认不做。
- `云端同步` 成本很高，也会引入账号、隐私、冲突合并问题；短期只保留本地导入导出和配置包。
- `主题系统` 不能只新增更多滑块，应该先固化 token 和预设，再决定是否开放深度编辑。
- 技术重构不能一次性拆散 4000 行 `app.js`，否则风险过高；应先按“可回滚的小模块”分阶段迁移。

## 不变约束

后续所有任务必须遵守以下约束：

- 保持 Chrome MV3 扩展能力。
- 保持零依赖、无构建。
- 页面模式 `newtab.html` 直接打开时仍能基本可用。
- 默认体验优先，深度配置靠后。
- 新增模块必须可关闭或可隐藏。
- 不把主页做成信息流、看板或复杂控制台。
- 不为了视觉效果引入大面积实时滤镜、常驻 `will-change` 或高频重绘。
- 新增浏览器权限前必须明确收益、替代方案和用户提示。

## 推荐版本路线

### V1.1 稳定收口

目标：把现有体验打磨稳定，为后续开发建立安全边界。

建议周期：1 到 2 周。

优先任务：

1. 建立手动验收清单
   - 桌面宽屏。
   - 窄窗口。
   - 移动端宽度。
   - 扩展模式。
   - 直接打开 `newtab.html` 的页面模式。
   - 亮背景、暗背景、高细节背景。

2. 收紧视觉 token
   - 统一表面层：背景、边框、阴影、圆角。
   - 统一强调色：主按钮、焦点态、选中态、拖拽态。
   - 统一文本层级：时间、日期、搜索、面包屑、书签标题、设置项。
   - 统一动效强度：hover、focus、modal、dock 拖拽。

3. 稳定搜索区
   - 搜索框 hover 和 focus 不应污染下方内容。
   - 搜索建议层级应稳定，不被 Dock、书签、设置按钮干扰。
   - 搜索引擎菜单和建议列表使用同一套表面规则。

4. 稳定书签卡片
   - 明确 favicon 尺寸、圆角、留白、失败 fallback。
   - 文件夹图标和站点图标视觉风格统一。
   - 编辑态按钮不挤压卡片，不改变网格尺寸。

5. 优化设置侧栏
   - 设置面板从“调参面板”收敛成“产品设置”。
   - 将滑块分组和预设关系讲清楚。
   - 保留高级项，但避免默认把所有参数都堆在第一屏。

交付标准：

- 默认主题稳定。
- 主要交互无明显视觉伪影。
- 桌面、窄窗口、移动端布局都不溢出。
- `README.md` 同步说明当前功能和验收方式。

### V1.2 结构整理

目标：在不引入构建工具的前提下，拆出清晰边界，降低继续维护 `app.js` 的成本。

建议周期：1 到 2 周。

原则：

- 不一次性大拆。
- 每次只迁移一个独立职责。
- 迁移后行为不变。
- 仍用普通 `<script>` 加载，不引入打包。

推荐拆分顺序：

1. `src/config.js`
   - `Config`
   - 默认链接
   - 壁纸来源
   - 搜索引擎
   - 样式预设

2. `src/i18n.js`
   - `I18N`
   - `t()`
   - 文案 key 整理

3. `src/storage.js`
   - `StorageManager`
   - `chrome.storage.local` 与 `localStorage` 兼容逻辑
   - 配置导入导出

4. `src/bookmarks.js`
   - Chrome 书签读取。
   - 书签编辑 API 封装。
   - HTML 书签导入解析。
   - 自定义图标映射。

5. `src/search.js`
   - 搜索历史。
   - 书签搜索索引。
   - 搜索建议。
   - 快捷命令解析。

6. `src/ui.js`
   - 时间日期。
   - 背景切换。
   - 书签渲染。
   - Dock 渲染。
   - 设置面板渲染。

样式整理顺序：

1. 先在 `newtab.html` 内重新划分 CSS 区块。
2. 再考虑拆到 `styles/base.css`、`styles/components.css`、`styles/responsive.css`。
3. 拆 CSS 前先确保 Manifest 和页面模式都能正确加载相对路径。

交付标准：

- `app.js` 不再继续膨胀。
- 新文件仍通过静态脚本加载。
- 功能行为和拆分前一致。
- 关键模块有简短边界说明。

### V1.3 主题系统

目标：把当前样式滑块升级成真正可选的主题，而不是只提供参数集合。

建议周期：1 到 2 周。

推荐主题：

- `Focus`
  - 默认办公主题。
  - 低 glow、低透明、强可读性。

- `Glass`
  - 保留玻璃质感。
  - 控制 blur、阴影和高光强度。

- `Dense`
  - 适合大量书签。
  - 更小图标、更密网格、更低 hover 位移。

- `Immersive`
  - 更突出壁纸。
  - 自动提高文字保护和遮罩强度。

主题应控制：

- 表面透明度。
- 文本对比。
- 强调色。
- 阴影密度。
- 动效力度。
- 图标尺寸。
- 网格密度。
- 背景遮罩。

不建议：

- 短期新增大量颜色滑块。
- 把主题做成纯 CSS class 但仍依赖散落 JS 参数。
- 让用户必须理解 blur、overlay、hue 等技术参数才能得到好看的页面。

交付标准：

- 至少 3 套主题可直接切换。
- 主题切换后搜索、书签、Dock、设置侧栏视觉一致。
- 保留高级滑块，但默认入口不抢占主题选择。
- 导出配置包含主题信息。

### V1.4 高频功能增强

目标：在不改变项目定位的前提下，补齐新标签页高频操作。

建议周期：2 到 3 周。

优先做：

1. 书签收藏夹置顶
   - 用户手动选择置顶文件夹。
   - 不新增浏览器权限。
   - 只影响当前页面展示，不修改 Chrome 书签结构。

2. 书签显示模式
   - 图标模式。
   - 图标加标题。
   - 紧凑模式。
   - 按文件夹单独记忆或全局记忆，优先全局。

3. 搜索快捷命令
   - 保留 `/` 搜书签。
   - 增加 `g keyword`、`yt keyword`、`gh keyword`、`ai keyword`。
   - 命令配置跟随搜索引擎配置导出。

4. 多套 Dock 配置
   - 工作。
   - 常用。
   - 娱乐。
   - 切换入口保持轻量，不在主页增加复杂面板。

5. 配置包
   - 导出主题、Dock、壁纸、书签图标映射。
   - 支持导入时预览覆盖范围。
   - 暂不做云端同步。

谨慎评估：

- 最近访问。
- 常用网站自动推荐。

这两项需要新增 `history` 或 `topSites` 权限。只有在用户明确接受权限变化时再做，并且必须提供关闭开关。

暂不建议：

- 天气组件。
- 日历组件。
- 待办模块。
- 数据看板。
- 多栏信息流。

这些功能容易破坏当前“快速打开、快速搜索、快速进站”的定位。

## 权限策略

当前 `manifest.json` 只有：

- `bookmarks`
- `storage`

当前 host 权限主要用于搜索建议代理：

- `https://suggestqueries.google.com/*`
- `https://suggestion.baidu.com/*`

后续新增功能时按以下规则处理：

| 功能 | 可能权限 | 建议 |
| --- | --- | --- |
| 书签收藏夹置顶 | 无新增 | 可做 |
| 书签显示模式 | 无新增 | 可做 |
| 搜索快捷命令 | 无新增 | 可做 |
| 多 Dock 配置 | 无新增 | 可做 |
| 主题导入导出 | 无新增 | 可做 |
| 最近访问 | `history` | 暂缓，需用户确认 |
| 常用网站自动推荐 | `topSites` 或 `history` | 暂缓，需用户确认 |
| 天气 | 外部 API host 权限 | 暂不做 |
| 云端同步 | 账号、网络、冲突处理 | 暂不做 |

## 技术债处理清单

### 高优先级

- `app.js` 职责过多，需要按配置、存储、书签、搜索、UI、拖拽逐步拆分。
- `newtab.html` 样式过长，需要先重排 CSS 区块，再考虑拆 CSS 文件。
- `I18N` 文案散在 `app.js` 内，后续新增功能会越来越难维护。
- 样式预设和 CSS token 的边界需要统一，避免 JS 变量和 CSS 变量重复定义。

### 中优先级

- 为 favicon 失败、缓存、fallback 建立更明确的策略。
- 为扩展模式和页面模式分别记录功能可用性。
- 为导入配置增加版本号和迁移逻辑。
- 为移动端 Dock 拖拽补充更明确的状态反馈。

### 低优先级

- 自动背景分析。
- 壁纸方案包。
- 复杂布局编辑器。
- 云端同步。

## 近期可执行任务清单

建议按以下顺序开工：

1. 写一份 `doc/QA_CHECKLIST.md`
   - 明确每次发布前怎么手动验收。
   - 覆盖桌面、移动、扩展模式、页面模式、不同背景。
   - 状态：已完成。

2. 整理视觉 token
   - 在 `newtab.html` 顶部重命名并归类 CSS 变量。
   - 删除重复或语义不清的变量。
   - 确定按钮、卡片、面板、弹窗共用规则。

3. 调整设置侧栏信息架构
   - 第一层是主题和壁纸。
   - 第二层是书签、Dock、搜索。
   - 高级外观参数收进高级区。

4. 稳定书签图标系统
   - 明确 favicon 加载顺序。
   - 明确失败后的 fallback。
   - 明确自定义图标、文件夹图标、站点图标的尺寸规则。
   - 状态：进行中，已补充候选去重、低质量图标过滤和统一 fallback 样式。

5. 拆出 `src/config.js` 和 `src/i18n.js`
   - 只迁移常量和文案。
   - 不改行为。
   - 验证扩展模式和页面模式都可加载。

6. 实现主题系统第一版
   - Focus、Glass、Dense。
   - 先替代现有 `balanced`、`compact`、`immersive` 的展示方式。
   - 保留原始滑块作为高级设置。

7. 实现书签显示模式
   - 只改展示，不改书签数据结构。
   - 配置进入导出文件。

8. 实现搜索快捷命令
   - 在现有搜索逻辑上扩展。
   - 不新增权限。
   - 命令规则写入 README。

## 验收标准

### 视觉验收

- 时间和搜索在亮色、暗色、高细节背景下都清晰。
- 搜索框 hover、focus、建议展开时不污染下方内容。
- 书签卡片第一屏主次明确，不因图标加载失败产生明显跳动。
- Dock 存在感足够，但不抢时间和搜索的视觉焦点。
- 设置侧栏像产品设置，而不是开发调参面板。
- 桌面、窄窗口、移动端无文本溢出和控件重叠。

### 技术验收

- 扩展模式可加载。
- 页面模式可打开。
- 配置导入导出可用。
- 书签同步、编辑、新增、删除可用。
- Dock 拖拽排序可用。
- 搜索建议失败时不影响基本搜索。
- 不引入依赖和构建步骤。
- 不新增权限，除非该版本目标明确要求。

### 文档验收

- `README.md` 反映当前功能。
- 新增权限必须在 README 中说明。
- 新增配置必须说明是否会被导出。
- 新增主题或命令必须给出用户可理解的使用方式。

## 暂缓事项

以下事项不是永久不做，而是不适合当前阶段：

- 云端同步。
- 天气、日历、待办。
- 数据看板。
- 信息流。
- 自动读取最近访问。
- 自动推荐常用网站。
- 复杂布局编辑器。

暂缓原因：

- 会显著扩大权限、隐私或维护成本。
- 容易破坏新标签页的轻量定位。
- 当前代码结构还没有准备好承接这些扩展。

## 结论

后续开发应按这个顺序推进：

1. 先稳定默认体验。
2. 再拆清代码结构。
3. 再做主题系统。
4. 最后补高频功能。

最适合下一个版本的目标是 V1.1：视觉和交互稳定收口。等 V1.1 完成后，再进入 V1.2 的小步重构。这样既能保住当前项目“轻量、好改、可直接加载”的优势，也能避免后续每加一个功能都牵动整页样式和全局状态。
